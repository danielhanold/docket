# Design: extract inline board rendering into a deterministic script (change 0022)

**Status:** design (brainstormed 2026-06-18)
**Change:** 0022
**Unblocks:** 0023 (consumes the shared helper + `resolve_deps`), 0024 (deterministic render is its premise).

---

## 1. Context

The `inline` board surface (`BOARD.md`) is regenerated by the **model** from the
prose algorithm in `docket-status`'s Board pass. It is pure, judgment-free
transformation over change-file frontmatter, yet it runs on the model's tokens
every `docket-status`, every `docket-implement-next` Step 0, and after every
status write — and is not byte-deterministic run-to-run (the reason the
board/source-drift check exists). The `github` surface was already extracted to
`scripts/github-mirror.sh` (ADR-0007); `inline` is the analogous extraction, and
its rendering algorithm is already fully specified in the SKILL's *Board →
Structure* section.

This change ships three artifacts: the **shared helper** (whose interface 0023
and 0024 depend on), **`render-board.sh`**, and the migration of
`github-mirror.sh` onto the helper so no parser is duplicated.

## 2. The shared helper — `scripts/lib/docket-frontmatter.sh`

The contract 0023/0024 build against. A sourced bash library, no side effects on
source. Mock seam `GIT="${GIT:-git}"` for the git-touching part only.

**Frontmatter accessors** (lifted **verbatim** from `github-mirror.sh`, which then
deletes its private copies — §5):

- `field FILE KEY` → first matching top-level scalar in the frontmatter, trimmed.
- `list_field FILE KEY` → `[a, b]` rendered as space-separated `a b` (empty for
  `[]`/unset).
- `has_section FILE STR` → exit 0 iff the body contains the literal line `STR`.

**Dependency resolution** — `resolve_deps CHANGES_DIR`:

- Scans `CHANGES_DIR/active/` + `CHANGES_DIR/archive/` **once**, building an
  id→status map, then resolves every change's `depends_on`.
- **Populates three global associative arrays**, keyed by integer id (the
  `STATUS_OF` array name already exists in `github-mirror.sh` — same idiom):
  - `STATUS_OF[id]` — the change's own status.
  - `DEP_STATE[id]` — `clear` | `waiting`.
  - `DEP_REASON[id]` — `""` when clear, else the worst unmet reason
    (`needs your merge` > `not yet built`; target `done` ⇒ satisfied, `implemented`
    ⇒ `needs your merge`, anything else/missing ⇒ `not yet built`).
- This makes the convention's "dependency resolution computed **once**, consumed
  by the board and the health checks" invariant literal: one parser, one resolver,
  sourced by `render-board.sh` (0022), `board-checks.sh` (0023), and
  `github-mirror.sh` (§5).

**`readiness CHANGES_DIR_FILE`** — the single readiness classifier, precedence
**pinned** (matches `github-mirror.sh`'s `readiness_label` exactly; only meaningful
for `proposed`):

1. `DEP_STATE[id] == waiting` ⇒ `waiting/<reason>` (**takes precedence over a
   missing spec** — this is the previously-ambiguous rule, now fixed: a stub that
   also waits renders as waiting).
2. else no `spec:` and not `trivial: true`: `auto-groom-blocked` if the body has
   `## Auto-groom blocked`, else `needs-brainstorm`.
3. else `build-ready`.

## 3. `render-board.sh` — the renderer

**CLI:** `render-board.sh --changes-dir DIR [--repo OWNER/REPO]`

- Reads `DIR/active/` + `DIR/archive/`; sources `lib/docket-frontmatter.sh`;
  calls `resolve_deps DIR` once.
- **Emits `BOARD.md` to stdout** — no git writes (caller redirects + commits, same
  discipline as `github-mirror.sh`). Deterministic and idempotent: same change
  files ⇒ byte-identical output.
- `--repo` builds `pr:`/`issue:` hyperlinks; defaults to deriving the slug from
  `$GIT -C DIR remote get-url origin`. **Offline** — no `gh`, no network.

**Output = the SKILL's *Board → Structure*, with the fiddly bits pinned here so a
golden test can lock them byte-for-byte:**

- **Count line** in fixed order: in progress · proposed · blocked · deferred ·
  implemented · done · killed — each emitted **only when count > 0**.
- **Emoji map:** 🟢 in progress · 🟡 proposed · 🔴 blocked · ⚪ deferred ·
  🔵 implemented · ✅ done · 🗑️ killed.
- **Sections** in that same status order; a section is **omitted when its count is
  0**. Within a section, rows sorted by **ascending numeric id**. The archive
  `<details>` sorts by **date descending, then id descending** (newest first).
- **Per-status columns:**
  - in progress: `# | Title | Priority | Spec | Branch`
  - proposed: `# | Title | Priority | Readiness` (cell from `readiness`, §2)
  - blocked: `# | Title | Priority | Blocked by`
  - deferred: `# | Title | Priority`
  - implemented: `# | Title | Priority | PR`
  - archive: `# | Title | Merged` (the archive-filename date)
- **Mermaid** `graph TD`: active nodes/edges first (id order — a change with
  `depends_on` emits one `parent --> child` line per dep; one with none emits a
  bare `  id`), then `done` nodes as `  id:::done` (id order), then the trailing
  `  classDef done fill:#d3f9d8;`. Exact serialization is locked by the golden
  fixture.
- **No generated-at timestamp** (counts convey freshness; a timestamp would churn
  every run).

## 4. Frontmatter parsing — stays hand-rolled (yq: no)

These passes read only flat top-level scalars and single-line flow lists; the
`field`/`list_field` helpers already cover 100%. `yq` cannot parse
markdown-with-frontmatter natively and its wins target a nested shape this data
lacks. The only nested parser is `sync-agents.sh`'s `agents:` config — **0018's**
scope, decided "keep as-is." So 0022 stays hand-rolled and is `related` to 0018,
not `depends_on`.

## 5. `github-mirror.sh` migration (lands here)

`github-mirror.sh` is the *source* of the verbatim helpers, so the extraction
must not leave two copies:

- Delete its private `field`/`list_field`/`has_section`; `source` the helper.
- Replace its inline `readiness_label` dep-walk with a call to the shared
  `resolve_deps` + `readiness` (the label-string mapping stays local to the mirror).
- `tests/test_github_mirror.sh` must still pass unchanged — pure refactor, no
  behavior change.

## 6. `docket-status` wiring (SKILL edit)

In `docket-status`'s **Board pass → `inline` surface**: replace the hand-render
prose with "invoke `scripts/render-board.sh --changes-dir <metadata tree>
> BOARD.md`." **Keep unchanged**: when the pass runs, the **separate-commit**
discipline, and the **regenerate-don't-3-way-merge** conflict rule (now literally
"re-run the script"). The dependency-resolution prose stays as the human-readable
spec of what the helper computes. Mode handling is unchanged — the skill already
points the metadata tree at `.docket/` (docket-mode) or the primary tree
(main-mode); the script only needs the dir.

## 7. Scope

**In scope:** `lib/docket-frontmatter.sh` (incl. `resolve_deps`, `readiness`);
`render-board.sh`; `github-mirror.sh` migration; the `docket-status` `inline`-pass
edit; `tests/test_render_board.sh` + a helper unit test for `resolve_deps`.

**Out of scope:** the merge sweep & health checks (0023); board/source-drift
retirement (0024); the `github` surface's behavior (unchanged); `yq` (0018);
changing the board's visual structure or readiness vocabulary (faithful
re-implementation only).

## 8. Test plan

- `tests/test_render_board.sh` — a fixture `changes/` tree spanning every status
  (in-progress, proposed build-ready / needs-brainstorm / waiting / auto-groom
  blocked, blocked, deferred, implemented, done, killed) plus `depends_on` edges,
  rendered and compared **byte-for-byte to a committed golden `BOARD.md`**. A
  second assertion: re-running on the same tree yields identical bytes
  (idempotence).
- Unit-test `resolve_deps`: assert `DEP_STATE`/`DEP_REASON` for satisfied,
  `needs your merge`, and `not yet built` targets, and a `depends_on`-clear change.
- `tests/test_github_mirror.sh` green after the §5 migration (regression gate).
