# Design — #0066 Auto-groom's critic re-check must be foreground; a forked skill that yields returns a half-done run

- **Change:** 0066 · `auto-groom-critic-recheck-foreground`
- **Date:** 2026-07-12
- **Author:** `docket-groom-next` (interactive groom). `superpowers:brainstorming` was unavailable in the session, so the brainstorm degraded to the convention's `auto` fallback (inline design) per the *Skill layer* missing-skill rule. The two open questions were settled to the **recommended** default each; every choice is recorded in *Design decisions* for a human to audit (clear `spec:` to re-groom if a call would have gone the other way).
- **Related:** #17 (subagent composition wiring), #61 (`context: fork` dispatch), #65 (the groom this was observed during) · ADR-0006, ADR-0017, ADR-0024

## Problem

On 2026-07-12, while `docket-auto-groom` (running as a `context: fork`) groomed #0065, it drafted a spec, dispatched its critic, got a *wrong-but-fixable* verdict, and entered the bounded revision round. It then **dispatched the critic's re-check as a background task and yielded**, returning to its caller with "Holding for the critic's re-check verdict — the monitor will notify me the moment it lands." To the parent the Skill tool reported `completed (forked execution)`: the run **looked finished**. It was not — the agent was alive mid-step-5, a 22 KB spec sat **uncommitted and unlinked** in the shared `.docket` worktree, and the stub still read needs-brainstorm. The parent inspected the "aftermath", concluded the fork had crashed, and committed the agent's working-tree files itself. Two writers were live in one worktree, each believing the other was gone. Nothing was lost — the landed content happened to be byte-for-byte the intended output — but that was luck.

Two independent defects, both load-bearing for autonomous operation:

1. **The skill's foreground rule doesn't cover the re-check.** `docket-auto-groom` §3 (Critic pass) qualifies the *initial* critic dispatch as `(foreground, …)`, then describes the revision round — "the critic re-checks only the revised items" — with **no foreground qualifier**. The agent read the gap literally and backgrounded the re-check.
2. **A forked skill has no channel to receive a notification, so "wait for a notification" is a yield.** A fork cannot be resumed by the task-notification it is waiting on the way a main-loop session can; awaiting one hands control back to the caller. This is the same family as the known SDD hazard (an implementer that backgrounds the long test suite stalls the loop) and it **generalizes beyond auto-groom**: any forked/dispatched skill that awaits a background child returns a half-done run to a caller that will believe it finished.

Left alone, an overnight drain silently hands build-ready-looking stubs (or uncommitted specs) to whatever runs next.

This change is **skill + convention wording plus one guard test** (and one dated ADR update note). It touches no script, no schema, and no dispatch *mechanics* — only the prose that governs how a dispatch is awaited.

## The observed facts (inputs, not decisions)

Taken as given by this design:

- A forked skill (`context: fork`) returns to its parent as a Skill tool result — `completed (forked execution)` — the instant it *yields*, which is indistinguishable in the parent from a real completion (this is the documented fork opacity of ADR-0026 / #0065).
- ADR-0024 established the **fork-exclusion principle**: a forked subagent "has no channel to the human" (Claude Code withholds `AskUserQuestion`, `EnterPlanMode`, … from subagents). The same absence applies to the **task-notification channel** — a fork awaiting a notification is not resumed by it; it yields.
- The convention's *Composition* paragraph (change 0017) already declares every dispatch **foreground** — "the parent suspends until the child returns" — and states the dispatch **contract is git state** on `origin/docket`, "re-read after a re-sync — never an in-context return." The rule existed; it was under-stated in one place (auto-groom's second round) and never spelled out as *the parent must actively block, never yield*.
- `docket-implement-next` (→ `docket-status`, `docket-adr`) and `docket-finalize-change` (→ `docket-rebase-resolver`, `docket-integration-repair`) dispatch **single-shot** children, each already qualified `(foreground, …)` at its site. There is no "second round" to under-qualify (verified against both SKILL.md files at design time).
- The `agents/docket-*.md` wrappers are **hand-authored source** (not generated by `sync-agents.sh`), each carrying its own one-line abort-and-report boilerplate; every wrapper injects `docket-convention` via `skills:`, so a rule stated in the convention reaches all of them with no per-wrapper edit.

## Design

Three edits — one closes the concrete gap, one states the general rule at the single contract source, one guards both — plus a dated pointer on ADR-0024.

### 1. Close the wording gap in `docket-auto-groom` §3 (Critic pass)

The revision-round clause is today (line 44):

> … **wrong but fixable from available context** (designer revises; ONE bounded revision round; the critic re-checks only the revised items) …

Qualify the re-check explicitly, in the same sentence that bounds the round, and point at the general rule so no reader can take the `(foreground, …)` on the *first* dispatch as applying only there:

> … **wrong but fixable from available context** (designer revises; ONE bounded revision round; the critic re-checks only the revised items — **this re-check is dispatched foreground exactly like the first pass: the designer blocks on the critic's return and never backgrounds it to await a notification, per the convention's *Composition* never-yield rule**) …

Wording only; the gate, its verdict vocabulary, and the one-round bound are untouched (explicitly out of scope in the stub).

### 2. State the general never-yield rule where it binds every dispatched skill — `docket-convention` *Composition*

The convention's *Composition (change 0017)* paragraph is the single contract source that every wrapper loads. Extend its existing "these dispatches are **foreground** (the parent suspends until the child returns)" with the rule the incident proves was needed — the parent must **actively block**, never yield — and the reciprocal caller-side reading (see §3, Design decision D2):

> **Foreground means the parent *actively blocks* on the child's return — it may never background a dispatched or forked child and *yield* to await a task-notification.** A forked/subagent skill has no channel to receive one (the same no-channel fact ADR-0024 states for the human), so "wait for the notification" hands control back to the caller and returns a **half-done run that the caller reads as `completed`**. Blocking on a child is the contract; yielding to wait for one violates it. Reciprocally, a caller must **not** read a bare `completed` as proof the child finished: it verifies the child's git-state transition (re-read after a re-sync, above) and **never adopts or commits a child's uncommitted working-tree files** on its behalf.

This binds `docket-auto-groom` (the re-check), the two single-shot dispatchers, and any future multi-round dispatch — one statement, at the contract altitude, no per-skill duplication (Design decision D1). The single-shot dispatchers need no edit: their sites are already `(foreground, …)` and single-shot, and the general rule now covers them by construction.

### 3. Guard both — `tests/test_composition_wiring.sh`

The suite that already owns the composition contract gains two positive-anchor doc sentinels (sampling, not parsing — LEARNINGS #5/#13; anchored on the *meaningful framing*, not incidental wording):

- **Convention states the never-yield rule** — e.g. `grep -qi "never background" "$CONV"` and that the paragraph forbids yielding to await a notification (anchor on `never .*yield` / `task-notification`). Deleting the new sentence flips this red.
- **Auto-groom's re-check is foreground** — anchor on the distinctive phrase committed in §1 (`re-check is dispatched foreground` / `foreground exactly like the first pass`). A bare count of "foreground" is rejected as fragile (the initial dispatch already contains one); the phrase pins the *second* qualifier specifically.

### 4. ADR-0024 — a dated `## Update` note (Design decision D4)

ADR-0024's decision text is immutable, but the convention sanctions a dated `## Update` note for a non-reversing context extension (the exact move #0065 made). Add one, extending 0024's "no channel to the human" to its corollary:

> `## Update — 2026-07-12` — the no-channel principle extends beyond the human: a forked/subagent skill also has no channel to receive a **task-notification**, so it may never background a child and yield to await one (that returns a half-done run reading as `completed`). The rule is stated in `docket-convention`'s *Composition* paragraph and enforced in `docket-auto-groom` §3. See #0066.

Recording it as an ADR update — not a new ADR — is deliberate: this is an **elaboration** of an Accepted decision, reversing and superseding nothing (Design decision D4). Listing `24` in the change's `adrs:` is load-bearing, not bookkeeping: terminal close-out publishes the `Accepted` ADRs named in `adrs:` onto `main`, so listing 24 is what carries the new note there (the #0065 lesson — do not "tidy" 24 out).

## Design decisions

Each open question was settled to the recommended default; recorded here for audit. The brainstorm was inline (no `superpowers:brainstorming` in-session), so these are the groomer's calls, not a human's.

**D1 — The general rule lives ONCE in the convention's *Composition* paragraph; only `docket-auto-groom` §3 gets a concrete wording fix. (Stub open question 1.)**
*Chosen:* state the never-yield rule at the single contract source every wrapper loads, and fix the one skill with an actual under-qualified round. *Rejected:* (a) **belt-and-suspenders** — also add a `(foreground)` qualifier at every dispatch site in `docket-implement-next` and `docket-finalize-change`; redundant, and it re-introduces the drift risk the model/effort-tier guard exists to prevent (the same fact restated in N places). (b) **fix auto-groom wording only** — narrowest, but the hazard recurs the moment any skill gains a multi-round dispatch, and it leaves the *general* lesson (the load-bearing half) unwritten. *Why:* the single-shot dispatchers were checked (stub open question 1) — their sites are already `(foreground, …)` and have no second round to under-qualify, so the general convention rule binds them by construction with zero per-skill edits. One contract statement + one concrete fix is the DRY option that still closes the actual gap.

**D2 — The caller-side guard is advisory, leaning on the existing git-state contract — not a hard, enforced check. (Stub open question 2.)**
*Chosen:* add a reciprocal *reading* rule to the same convention sentence — a caller must not treat a bare `completed` as proof of completion; it verifies the git-state transition (the contract already stated in the paragraph) and never adopts a child's uncommitted files. *Rejected:* (a) **hard guard** — treat a groom that returns without either a linked spec/trivial verdict or an abstain section as a failed run, enforced; stronger, but it adds standing machinery and a *second* home for the completion-contract (the git-state contract already is that home), and it only detects the symptom the root-cause fix (§1+§2) removes. (b) **no guard** — rely solely on never-yield; but the incident's damage came from the *caller* acting on a half-done tree, and one sentence closes that reading gap at no machinery cost. *Why:* the root cause is the yield; §1+§2 remove it. The residual — a caller misreading `completed` — is closed by pointing at the git-state contract that already exists, not by building a new validator. Advisory is the proportional fix; if premature returns ever recur *despite* the never-yield rule, a hard guard is a clean follow-up change.

**D3 — Exit is a spec, not a trivial verdict.**
*Chosen:* spec. *Rejected:* `trivial: true` — tempting because the diff is a few sentences. *Why:* the change carried two genuine design questions (rule home / scope, guard strength) with real rejected alternatives; that is precisely what a spec, not a trivial verdict, is for. The small diff is downstream of settled design, not evidence the design was mechanical.

**D4 — Record the general principle as a dated `## Update` on ADR-0024, not a new ADR; and not nothing.**
*Chosen:* a two-sentence dated note on 0024 extending its no-channel principle to task-notifications, with `24` listed in `adrs:` so close-out publishes it to `main`. *Rejected:* (a) **a new ADR** — this reverses and supersedes nothing; it elaborates 0024's own principle, and an ADR ledger of elaborations erodes into a changelog. (b) **no ADR touch at all** — cheaper, and defensible since the convention paragraph is the true contract home; but a reader landing on ADR-0024 (the fork-exclusion decision) would never learn the no-channel fact also forbids notification-yields, and the convention's `## Update` mechanism exists for exactly this. *Why:* the note is the lowest-drift way to make the principle discoverable from the decision that spawns it, at the cost of two sentences. **This is the one element beyond the stub's literal "What changes"; it is pure-addition and pure-subtraction to drop** — a human who wants the change kept to skill+convention+test can strike D4 and `adrs:` with no effect on §1–§3.

## Open questions

None outstanding. Both stub open questions are resolved above — D1 (same gap in the single-shot dispatchers: checked, none; general rule covers them) and D2 (guard advisory, leaning on the git-state contract) — and recorded as audited decisions.
