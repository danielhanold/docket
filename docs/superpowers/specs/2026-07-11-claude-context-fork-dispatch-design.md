# Spec — Claude Code skill-invocation parity via `context: fork`

**Change:** 0061
**Date:** 2026-07-11
**Status:** approved (brainstormed with the human, 2026-07-11)

## Problem

docket ships model/effort-pinned subagent wrappers for its autonomous skills (`agents/docket-*.md`, generated by `sync-agents.sh`). The pin is honored when a skill runs as a **dispatched** subagent, but **not** when a skill is **invoked directly**.

`sync-agents.sh` (~line 212, `HARNESS_HAS_DISPATCH_RULES`) encodes the assumption that *"only cursor exhibits the inline quirk"* — that only Cursor runs a directly-invoked skill at the current session model. **This is false.** Claude Code does the same: when a human runs `/docket-status` (or Claude auto-invokes the skill), it executes **inline in the main session at the session model**, defeating the skill's model/effort pin (e.g. `docket-status` is pinned to `haiku/medium`, `docket-implement-next` to `opus/xhigh`).

Cursor solves this with a generated `alwaysApply` dispatch rule (`.cursor/rules/docket-dispatch.mdc`) that forces a `Task` dispatch to the pinned wrapper. Claude Code has a **native** mechanism that did not exist / was not accounted for when the assumption was baked in: the `context: fork` skill frontmatter field.

## Approach

Add two frontmatter lines to the committed `SKILL.md` of each headless-safe autonomous skill:

```yaml
context: fork
agent: docket-<name>   # routes the fork into the existing pinned wrapper
```

Claude Code honors this: invoking the skill runs it in an isolated subagent of type `docket-<name>` — the existing pinned wrapper — so the wrapper's `model` / `effort` apply. `agent:` is required; a bare `context: fork` would fork into an unpinned default agent.

Because skills are a single shared source symlinked into every harness's skills dir (`link-skills.sh`), and unknown frontmatter is inert elsewhere, the same `SKILL.md` is safe cross-harness: Claude Code honors `context: fork`; Cursor ignores it and keeps its own dispatch rule; Codex/others ignore it. **No `sync-agents.sh` generation, no hooks, no CLAUDE.md routing.**

### Selection — fork only human-non-interactive skills

A forked subagent has **no channel to the human**: Claude Code documents that `AskUserQuestion`, `EnterPlanMode`, `ExitPlanMode` (unless `permissionMode: plan`), `ScheduleWakeup`, and `WaitForMcpServers` are unavailable to subagents "even when listed in the `tools` field." So a skill that must prompt the human mid-run cannot be forked.

| Skill | Needs the human mid-run? | Fork? |
|---|---|---|
| `docket-status` | no | ✅ `agent: docket-status` |
| `docket-adr` | no (writes to `docket` branch; main-publish is deferred to finalize) | ✅ `agent: docket-adr` |
| `docket-implement-next` | no (stops **at** the human merge gate; never merges) | ✅ `agent: docket-implement-next` |
| `docket-auto-groom` | no (abort-and-report, never prompts) | ✅ `agent: docket-auto-groom` |
| `docket-finalize-change` | **yes** — merge confirmation | ❌ excluded — see below |
| `docket-new-change` | **yes** — brainstorm dialogue | ❌ excluded |
| `docket-groom-next` | **yes** — brainstorm dialogue | ❌ excluded |

The 4 forked skills already run headless via their autonomous wrappers today, so they are **proven fork-safe**.

`docket-finalize-change` is excluded because a headless merge hits Claude Code's auto-mode **"Merge Without Review" soft-deny** classifier, which an autonomous subagent cannot clear (no human to express merge intent). This is a Claude-Code-specific classifier — Cursor lacks it, which is why Cursor *does* dispatch finalize; the difference is legitimately per-harness, not a docket inconsistency. Making finalize forkable/autonomous is tracked separately as **change 0062** (it is a permissions-policy decision, not a model-pin fix).

### Dispatch-only helper agents — untouched

`docket-integration-repair`, `docket-rebase-resolver`, `docket-auto-groom-critic`, `docket-brainstorm-consultant` have **no `SKILL.md`** — they are invoked only via `Task(subagent_type: …)`. A human can never invoke them as a skill, so the inline quirk cannot touch them; their pins already apply on every dispatch. No change.

### Composition — no recursion

The wrappers preload the skill via `skills: [docket-<name>, docket-convention]` frontmatter (content injection at agent startup), **not** via a `Skill` tool call in the wrapper body. `context: fork` fires on **invocation**, not on preload, so the flow terminates:

```
/docket-status → Skill invocation → fork into docket-status agent (haiku/medium)
              → agent starts with the skill preloaded → runs it → returns.   [no re-fork]
```

### Correct the wrong assumption

- `sync-agents.sh`: fix the "only cursor exhibits the inline quirk" comment (~line 212). Claude Code exhibits it too; it is handled by the per-skill `context: fork` frontmatter (declarative, native), while Cursor uses the generated dispatch rule. Keep `HARNESS_HAS_DISPATCH_RULES` cursor-only — Claude does **not** need a generated dispatch *rule*; it uses the frontmatter mechanism instead.
- README harness/dispatch section: document the **two-mechanism story** (Cursor rule vs. Claude `context: fork`) and the **fork-exclusion principle** (fork only human-non-interactive skills, because subagents cannot reach the human).

## Verification (plan-time; all degrade gracefully)

1. **Fork + preload compose** without double-running the skill body, and the wrapper's model/effort pin applies to the fork. Empirical: run `/docket-status`, confirm it executes as a `docket-status` subagent at `haiku`.
2. **Inert cross-harness**: `context: fork` is ignored when the shared `SKILL.md` is symlinked into Cursor/Codex/etc.
3. **Graceful on old Claude Code** (< v2.1.117, which predates forked subagents): unknown frontmatter is ignored → the skill runs inline = today's behavior, strictly no worse.
4. **Nested fork**: when `docket-implement-next`'s subagent invokes `docket-adr` / `docket-status` (which now carry `context: fork`), it either no-ops to inline within the already-pinned subagent or forks harmlessly ("a fork can't spawn further forks" → degrades to inline, which is already at the right model). Confirm no breakage.
5. **Human-path headless-safety**: confirm the human-invoked path of each of the 4 forked skills has no interactive branch that would silently die in a subagent (they are proven via the autonomous wrappers, but re-confirm the human entry point).

## Testing

- Add/extend a `sync-agents` (or lightweight structural) test asserting the **4** headless skills carry `context: fork` + a matching `agent: docket-<name>`, and the **3** interactive/excluded skills do **not**.
- Consider a `sync-agents.sh --check` advisory assertion for the same invariant, so a newly-added autonomous skill is not silently left un-forked (mirrors how the Cursor dispatch fragment warns on a missing fragment).

## Expected ADR

This records a genuine architecture decision — *Claude Code uses `context: fork` frontmatter as its inline-skill dispatch mechanism; fork only human-non-interactive skills* — paralleling ADR-0017/0020 (Cursor dispatch rule) and building on ADR-0008 (agent layer). The implementer is expected to record a new ADR at build time (docket-adr step).

## Out of scope

- Forking `docket-finalize-change` / autonomous merge — blocked on the auto-mode Merge-Without-Review classifier; **change 0062**.
- Any change to the Cursor dispatch-rule mechanism or to other harnesses.
- Consolidating or replacing the wrapper machinery with `context: fork` (the "consolidate" option was considered and declined in favor of the minimal parity fix).
