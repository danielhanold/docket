## Verify (human)

Automated coverage is comprehensive (50 test files green, incl. hermetic `tests/test_reclaim_claims.sh`), so these are judgment checks, not re-tests:

- [ ] **Before enabling `reclaim.auto: true` anywhere shared, understand the mutation path.** The ONLY path that mutates + pushes to origin is `docket-status` full pass → `reclaim_pass` (gated by BOTH a `[reclaimable]` finding AND `reclaim.auto: true`) → `reclaim-claims.sh`. It flips an expired-lease, **no-branch** change back to `proposed`. Default is off (detect + recommend only). Verified unreachable under `--board-only`.
- [ ] **Multi-machine `reclaim.auto: true` residual (spec §7-H).** The no-branch orphan guard reads *local* remote-tracking refs. On a single clone the `refs/heads/feat/<slug>` arm is authoritative; across machines, machine B could in principle reclaim machine A's live-but-unpushed build if its lease exceeded the TTL. Contained by: auto-off default, generous TTL, CAS re-read, and callers syncing first. If you set `reclaim.auto: true`, set it on the committed repo layer (consistent across clones), not per-machine.
- [ ] **Optional smoke of the self-heal** on a throwaway repo: create an `in-progress` change with a `claimed_at:` older than the TTL and no `feat/` branch, run `docket.sh reclaim-claims`, confirm it returns to `proposed` with `branch:`/`claimed_at:` cleared and `reconciled: false`.

## Findings

- **Reclaim CAS uses `fetch` + `reset --hard <remote>/<branch>` + re-read eligibility (bounded retry), NOT the `pull --rebase` replay used elsewhere in docket.** Discovered during Task 3: the plan's sketch re-read the just-written working tree, which always reads back our own flip (would spuriously report every race as "skipped" and leave a stale unpushed commit). Correct approach discards the local reclaim commit and re-derives from fresh origin reality — mirroring `docket-implement-next`'s own sanctioned claim-loop discipline (ADR-0004 final-push CAS). Safe because push-per-item means the local branch never carries more than the current item's one unpushed commit, so `reset --hard` only ever discards that. A genuine fetch/reset failure or loop exhaustion `die`s (never mislabels as a race). Documented in `scripts/reclaim-claims.md`. This is the first `reset --hard` in a docket CAS path (precedent elsewhere is `pull --rebase`) — flagged here in case a future reader wants it promoted to an ADR; it currently rides ADR-0004's principle + the script contract.
- **Reconcile refinements (spec §10):** facade op is **`reclaim-claims`** (op-name == script-basename is a hard facade invariant, `tests/test_docket_facade.sh`), not the spec's prose `docket.sh reclaim`; and `reclaim.lease_ttl` is an **integer count of hours** (default 72), mirroring `learnings.cap`'s bare-integer + fail-closed pattern, converted to seconds internally.
- **`[reclaimable]` producer/consumer contract.** `board-checks.sh`'s `stale-in-progress` finding ends in the literal marker `[reclaimable]` ONLY for the expired-AND-no-branch case; `docket-status.sh` keys on it with `grep -F`. The has-branch message `(not auto-reclaimable)` cannot false-match (brackets are load-bearing). Documented in `scripts/board-checks.md`.
- **`## Artifacts` is intentionally NOT regenerated by reclaim.** Reclaim mutates only non-link-bearing fields (`status`/`branch`/`claimed_at`/`reconciled`/`updated`), so per the convention's field-write rule the Artifacts block is untouched — this supersedes the spec §3-3 phrase "refreshes the Artifacts block."
- **Metadata-branch-invisible-to-suite:** reclaim operates on `docket`-branch change files, which the integration-branch suite cannot see; its behavior is proven by the hermetic `tests/test_reclaim_claims.sh` (`NOW`/`GIT` seams, real temp git repo + bare origin). No migration is required — a pre-migration change with no `claimed_at:` is never reclaimed (no positive evidence of expiry), verified end-to-end.

## Follow-ups

- **Branch adopt-or-supersede for the has-branch case (recommended new change).** An expired-lease change that HAS a pushed feature branch is deliberately never auto-reclaimed (it may carry real work; clearing `branch:` would orphan it and collide with the next claimant's `git worktree add -b`). It stays flagged by `stale-in-progress` for a human. Bringing branch adopt-or-supersede into the autonomous path is genuine judgment — a separate change (spec §7-C).
- **`#88` (loop continuation) and `#8` (parallel drain)** are made safer by this self-heal but are not implemented here.
- Consider promoting the reclaim CAS `reset --hard` decision (first Finding) to an ADR if the pattern recurs.
