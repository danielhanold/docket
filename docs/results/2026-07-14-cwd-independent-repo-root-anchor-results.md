# Anchor the repo root to the main worktree — results
Change: #75 · Branch: feat/cwd-independent-repo-root-anchor · PR: (opened at close-out) · Plan: docs/superpowers/plans/2026-07-14-cwd-independent-repo-root-anchor.md · ADRs: 34, 35

## Verify (human)

No interactive checks are required — every invariant is asserted by the suite (40 files, green, zero failures). Listed for the merge gate only because the central promise is destructive if it ever regresses:

- [ ] Cleanup refuses fail-closed from inside its target, remote branch intact — `bash tests/test_closeout.sh | grep "D1(inside target)"` (the load-bearing one is *"THE REMOTE BRANCH STILL EXISTS"*).
- [ ] The anchor is CWD-independent live: `docket.sh env` from the main tree, `.docket/`, and a subdirectory all print byte-identical `REPO_ROOT` / `METADATA_WORKTREE`. Confirmed during the build; re-checkable in one line.
- [ ] No nested `.docket/.docket` after a preflight from inside `.docket/` — `git worktree list` shows only the main tree, `.docket`, and feature worktrees.

## Findings

The change delivers all three verified defects — **D1** (cleanup deleted the *remote* branch then failed, from a linked-worktree CWD — partial irreversible data loss), **D2** (preflight minted a nested `<repo>/.docket/.docket`), **D3** (cleanup orphaned the agent's own CWD) — via one shared anchor (`scripts/lib/docket-root.sh`) adopted by the resolver, preflight, cleanup, docket-status, render-change-links, and the finalize skill. Two ADRs record the durable decisions: **ADR-0034** (root anchors to the main worktree; the three prohibitions; `REPO_ROOT` plain-only; the `git -C "$mw"` pathspec rule) and **ADR-0035** (teardown is fail-closed).

Six things the build surfaced beyond the spec — most found by the per-task or whole-branch review, each mutation-confirmed:

1. **The `docket-status.sh` landmine was ALREADY LIVE mid-branch, not dormant.** The spec framed the artifacts-refresh block (`sweep_execute_one`, ~line 379) as *dead today*, coming alive only when this change anchors `$mw`. In fact Task 3's preflight anchor already made `METADATA_WORKTREE` absolute, and `docket-status.sh`'s old `mw="${METADATA_WORKTREE:-.docket}"` simply read it — so the block was live, and its `return 0` on a failed push was really abandoning `terminal-publish` **and** `cleanup`, from commit `c42ae5b` onward. The Task 5 fixture's first pre-edit run proved it (the abandonment asserts were red while the block's own asserts were green). Fixed: the block now reports `sweep-failed <id> render-change-links commit-failed|push-failed` on the line-oriented channel and **continues** the close-out; a stale `## Artifacts` link block is cosmetic and self-heals, an orphaned worktree + unpublished record are not. Mutation-confirmed in both directions (restoring `return 0` reddens the abandonment asserts; reverting `$mw` to relative reddens the deadness asserts — they are complementary, neither alone suffices).

2. **A second D1-shaped path survived inside cleanup itself** (found in Task 4 review, finding 2). When the worktree target does not exist but the local branch is still checked out elsewhere (reachable via a hand-passed `--worktrees-dir` override), removal was skipped, `git branch -D` fell into `|| true`, and execution still reached `git push --delete`. Closed with a tight, principled invariant now in ADR-0035: **never delete the remote branch while the local branch still exists.** No production call site passes `--worktrees-dir`, so it was not a live regression — but it is the same half-destructive shape the change exists to abolish.

3. **A vacuous provenance test was sitting on a live data-loss instance** (Task 4 review, finding 1). The out-of-tree-refusal assert (`mkdir -p "$tmp/outside-wt"`) never created `outside-wt/wid`, so `[ -e "$target" ]` was false, the provenance guard was skipped, and the non-zero rc it observed came from the postcondition — while in that same fixture the script fell through and destroyed the remote branch. The guard whose name is about preventing data loss was reporting success over a live instance of it. Fixed to actually create the target and assert the remote survives; mutation-confirmed.

4. **Two brief/plan-authored code snippets carried real bugs** — the recurring "plan code is unvetted" hazard, hit five times across tasks. The worst: the plan's §5 fault injection (`git remote set-url --push` to a bogus target) would have broken *every* push in the fixture (worktrees share `.git/config`) and hung the suite in `archive-change.sh`'s `until push; do pull --rebase; done` loop. Replaced with a `pre-receive` hook on the bare origin that rejects only the `refresh artifacts links` commit on `refs/heads/docket` (range-checked, not tip-only). Others: a `pipefail` leak in `docket_main_worktree` (git's exit 128 propagating through the pipe — fixed with a trailing `return 0`); a mutation recipe that crashed on `set -u` before reaching its assert; a fixture editing a `.docket` path that does not exist until preflight creates it.

5. **Two pre-existing test fixtures declared an unemittable config shape** (`DOCKET_MODE=main` + `METADATA_WORKTREE=.docket`; the resolver emits `METADATA_WORKTREE=.` in main-mode). The old hard-coded `mw="."` masked it; routing through `docket_metadata_worktree` exposed it. Corrected the fixtures to the faithful shape — same asserts, no weakening.

6. **The `docket-status` SKILL prose contradicted the new report-line semantics** (Task 5 review, finding 1). After the block became report-and-continue, `skills/docket-status/SKILL.md` still told the agent that any `render-change-links` `sweep-failed` leaves the record unpublished and needs a manual `terminal-publish` — now false for `commit-failed`/`push-failed` (the close-out completed), and dangerously so under the repo's routine concurrent-loop push races. Narrowed to match the contract exactly. The prose-drift-restates-a-fact hazard from LEARNINGS, one level up.

**On guard discipline.** Every new assert was mutation-tested against the *real* tree, not only fixtures — the cleanup CWD refusal, the preflight nested-worktree guard, the `REPO_ROOT` plain-only emission (with its positive control proving the absence-assert can fire), the §5 report-and-continue paths, and the main-mode anchor. The whole-branch review independently re-ran each mutation and watched it redden. The whole suite (40 files) was run foreground after every task; it caught an out-of-goal regression (a wiring sentinel in `test_closeout.sh` reddened by an over-aggressive SKILL rewrite in Task 5's fix) that no task-scoped review would have — exactly what the enumerated-floor LEARNINGS rule predicts.

## Follow-ups

- **`ensure-claude-settings.sh:24` still resolves its root via `git rev-parse --show-toplevel`** — CWD-derived, so run from a linked worktree it would write `.claude/settings.local.json` into the wrong tree. It is a session-time install helper, not facade-reached, and explicitly out of 0075's scope (the spec references it only as the reason `REPO_ROOT` is plain-only). Not a regression; a reasonable follow-up change to complete the anchor's coverage.
- **Behavior change to state in any migration/upgrade note:** a resolver invoked from `<repo>/sub/` now reads `<repo>/.docket.local.yml` and targets `<repo>/.docket` (previously `<sub>/…`), and `--bootstrap` seeds `<repo>/.gitignore`. Intended (ADR-0034), pinned by test, and almost certainly what every caller already wanted — but it is a behavior change, not behavior-preserving.
- **The `git -C "$mw"` pathspec rule (ADR-0034 corollary) is currently unguarded.** A repo-wide grep during Task 5 found no other `$mw`-prefixed-relative pathspec site, but nothing prevents a new one. A structural sentinel over `git -C "$VAR" … -- <pathspec>` sites would make the rule enforced rather than documented.
