# terminal_publish opt-in default — results
Change: #0084 · Branch: feat/terminal-publish-opt-in-default · PR: <url> · Plan: docs/superpowers/plans/2026-07-16-terminal-publish-opt-in-default.md · ADRs: 40

## Verify (human)

- [ ] **This repo now publishes by explicit opt-in, not by default.** `.docket.yml` pins `terminal_publish: true`, read via `git show origin/HEAD:.docket.yml` — so the pin and the flip land **atomically at merge**, with no window where this repo silently stops publishing. Confirm you want that pin kept (the alternative is to let this repo stop mirroring records onto `main`).
- [ ] **The merge gate exercises the direct-push path.** Because this repo opts in, close-out will push the terminal record straight to `main` — expect the usual branch-protection / `--admin` friction.
- [ ] **Other repos you have migrated will silently stop publishing** on their next pull of docket's scripts, unless they add `terminal_publish: true` themselves. The migration posture is deliberately docs-only. The only runtime signal is a stderr line at each close-out (`terminal_publish: false — skipping publish onto <branch>…`), which does not distinguish "unset" from "explicit false".

## Findings

- **The four executable default-encoding sites were inventoried correctly; the prose sites were not.** Reconcile derived the code inventory by whole-repo grep (per the LEARNINGS enumerated-floor rule) and it held exactly: `docket-config.sh:199`, `terminal-publish.sh:30`, `docket-status.sh:389`, plus `tests/test_docket_status.sh` Case B. The final whole-branch review confirmed no fifth executable site exists.
- **The prose inventory was a floor, not the set — the fourth recurrence of that lesson here.** ~10 live doc/skill sites still framed publishing as unconditional. The root shape: **prose naming `main`-mode as the SOLE no-op exception**. That framing was *exhaustive and correct* before this change — the mode guard genuinely was the only unconditional no-op — and became exactly half the gate after it. Such lines contain neither "default" nor "true", so every keyword grep was structurally blind to them. Found only by semantic reading in review.
  - Worst three: `scripts/docket-status.md:130` quoted the literal `${TERMINAL_PUBLISH:-true}` expression this change *deletes*; its `swept` definition (L209) claimed a publish the branch's own Case B test proves does not happen; and `skills/docket-convention/SKILL.md:84` — an *instruction* — would have made a parent agent read a healthy opted-out run as a **failure**.
  - Reconcile wrongly cleared `scripts/docket-status.md`: the inventory grep used `--include="*.sh"` for the `TERMINAL_PUBLISH:-` pattern and case-sensitive lowercase `terminal_publish` for `.md` files, so an uppercase `${TERMINAL_PUBLISH:-true}` inside a `.md` fell through both. The spec *had* listed the finalize/status skills; reconcile overrode it on a bad grep.
- **A default flip can silently make a fence test vacuous.** `test_docket_config.sh`'s coordination-key fence blocks probed with `terminal_publish: false` and asserted the value "stays true". Once the default is `false`, the *ignored* value and the *default* coincide — asserting `= false` would pass whether or not the fence works. The fixtures now probe with `true` (the non-default); a reviewer mutation-test confirmed the assertions go red when the fence is defeated. **General rule: a fence/ignore test must probe with the NON-default value, or the flip that changes the default quietly guts it.**
- **A test whose premise the change deletes should be deleted, not re-gated.** `test_closeout.sh` had a block named *"back-compat — omitting `--enabled` still publishes (default true)"*. Re-gating it with `--enabled true` made it pass — as a byte-for-byte duplicate of the adjacent explicit-true block, under a name contradicting its own body. Deleted; omitted-flag coverage lives in `test_terminal_publish.sh`.
- **Case B was inverted, never deleted** — it guards a real `set -u` unbound-variable crash in `sweep_execute_one`, and the `:-` expansion *is* that guard. Only the fallback value flipped.
- **ADR-0040** records the flip (`relates_to: [27]`, `change: 84`). ADR-0027 stays `Accepted` and untouched by design — it decided *where* the knob may be set and *where* the gate lives, not the default value. Its Context still says the default is `true`; that is correct-as-history, which makes ADR-0040 the ledger's only statement of the new default.

## Follow-ups

- **`tests/test_docket_status.sh:946` — assertion name overstates what it proves.** `sweep exits zero (no unbound-variable crash)` cannot actually detect that crash: `sweep_execute` runs under process substitution (`docket-status.sh:473`), so a subshell death never reaches the outer exit code. The crash *is* caught, by the sibling `sweep emits swept` assertion — no coverage is lost. Pre-existing, not introduced here. Worth a rename so the name matches the mechanism.
- **`migrate-to-docket.sh` grants the terminal-publish push allow-rule unconditionally**, pre-authorizing a push the new default will never make. `README.md:570` now documents this honestly, but gating the grant on the opt-in would be least-privilege. Candidate change.
- **#0083's gap-detection health check** stays out of scope. This flip narrows its blast radius (far fewer repos publish at all) but does not replace it.
- **A publish-via-PR mechanism** remains unbuilt — the option for a repo that wants records on its code line without direct pushes. Currently such a repo must choose between direct commits and no records on `main`.
