## Why

When a change reaches a terminal state (or `docket-adr` publishes a standalone/
status-changed ADR), `terminal-publish.sh` copies the *individual* Accepted ADR
files onto the integration branch — but never the ADR index (`docs/adrs/README.md`).
The index is a derived view regenerated by `render-adr-index.sh` and committed on
`metadata_branch`; on the integration branch it is only ever whatever was last
hand-published. So as `done`/`killed` changes publish their ADRs to the
integration branch, the index there **silently drifts**: ADR files accumulate
without matching index entries, leaving the published index pointing at an old
high-water mark while newer ADR files sit unlisted.

Concrete evidence (dogfood repo `markhaus`, 2026-06-23): `main` carried ADR files
through 0051, but `docs/adrs/README.md` on `main` still stopped at ADR-0042 —
0043–0051 were present as files but absent from the index. It had to be
regenerated and pushed to `main` by hand (the same one-off had happened once
before, "publish index entries for ADR-0038–0042 to main"). A reader browsing
the integration branch sees an index that disagrees with the ADR files beside it.

## What changes

Make the integration-branch ADR index stay consistent with the ADR files
published there, automatically — no manual catch-up.

The likely shape (for the designer to confirm): whenever `terminal-publish.sh`
publishes one or more Accepted ADRs to the integration branch, also regenerate
`docs/adrs/README.md` from the integration branch's ADR files
(`render-adr-index.sh --adrs-dir <integration adrs>`) and include it in the same
publish commit. The index regenerated on the integration branch must reflect only
the ADR files actually present there (not the metadata branch's superset), so
every link resolves. This must cover both publish shapes — change-publish
(`--id`, when the change's `adrs:` include Accepted ones) and ADR-only publish
(`--adr`). A no-op in `main`-mode (no separate integration branch).

## Out of scope

- Publishing `BOARD.md` or other metadata-only derived views to the integration
  branch — those deliberately stay on `metadata_branch`.
- Changing which records terminal-publish copies (change file, spec, Accepted
  ADRs) beyond adding the index refresh.
- Back-filling already-drifted integration branches — that stays a manual
  regenerate (this change prevents future drift).

## Open questions

- Should the index refresh ride inside `terminal-publish.sh` (one publish commit
  with the ADR files + index), or be a separate step the calling skills invoke
  after publish? In-script keeps it atomic and covers every caller.
- When a publish copies an ADR whose change is `done` but a *later* ADR's change
  is not yet terminal, the integration index will trail the metadata index by
  design — confirm that "index reflects what's on the branch" is the intended
  contract (it is, to keep links resolvable).
