## Why

The `inline` board surface (`BOARD.md`) is currently regenerated by the
**model**, reading the prose algorithm in `docket-status`'s Board pass
(`## Board â†’ Structure (in order)`): parse every change's frontmatter, run the
dependency-resolution pass, then emit the count summary, the emoji-grouped
status tables with their readiness cells, the Mermaid dependency graph, and the
collapsible archive section.

That is a **pure deterministic transformation over the change files** â€” no
judgment is required at any step â€” yet it runs on the model's tokens on *every*
`docket-status` invocation, *every* `docket-implement-next` Step 0, and after
*every* status write (claim, reconcile-kill, `implemented`, sweep). It is the
single largest avoidable token cost in the routine loop.

Worse, model rendering is not byte-deterministic run-to-run, which is precisely
why docket carries a dedicated **"Board/source drift" health check** to catch a
writer that rendered the board inconsistently with the change files. A script
removes that whole failure class for `inline`.

The precedent already exists: the `github` board surface was extracted into the
deterministic, idempotent, mock-seamed `scripts/github-mirror.sh` (change 0011,
refined under 0020's progressive-disclosure pass). The `inline` surface is the
natural next extraction, and its rendering algorithm is *already fully
specified* in `docket-status`'s Board â†’ Structure section â€” it reads as a script
spec, not a prompt.

## What changes

To be finalized at brainstorm. The likely shape:

- Add **`scripts/render-board.sh`** â€” deterministic and idempotent (same change
  files â‡’ byte-identical `BOARD.md`), with the same contract style as
  `github-mirror.sh`: `--changes-dir DIR`, `--dry-run`, a mock seam, stderr
  logging, no git writes (the caller commits). It reads `active/` + `archive/`
  frontmatter, runs the shared dependency-resolution pass, and emits `BOARD.md`
  byte-for-byte per the documented Structure (count line, emoji-grouped tables,
  readiness cells including the `âŹł waiting on #N` and `needs-brainstorm` /
  `auto-groom blocked` rules, Mermaid graph with the `done` tint, archive
  `<details>`).
- Rewire `docket-status`'s Board pass so the `inline` surface **invokes the
  script** instead of hand-rendering â€” the skill keeps owning *when* to render
  and the commit/push discipline (separate commit, regenerate-don't-merge on
  rebase conflict); the script owns *how* to render.
- Factor the **dependency-resolution pass** into a shared core both
  `render-board.sh` and `github-mirror.sh` can consume (today it is prose
  consumed by the board, the sweep, and the health checks).
- Add **`tests/test_render_board.sh`** (fixtures â†’ expected `BOARD.md`),
  matching the existing `tests/test_github_mirror.sh` pattern.

## Out of scope

- **The merge sweep and the health checks** â€” `docket-status`'s other two
  passes. Whether *those* are scripted or stay model-driven is deliberately a
  separate decision (change 0023), because the sweep carries terminal-publish /
  harvest side effects and some checks involve judgment. This change touches
  only `inline` rendering.
- The `github` surface â€” already scripted (`github-mirror.sh`).
- Changing the board's visual structure, columns, or the readiness vocabulary.
  This is a faithful re-implementation of the documented Structure, not a
  redesign.
